    观察点¶

    在ReactiveX中，观察者订阅观察点。 然后该观察者对观察点发出的任何项目或项目序列作出反应。 这种模式有利于并发操作，因为它在等待观察点发送对象时不需要阻塞，而是以观察者的形式创建一个哨兵，随时准备在观察点将来的任何时间作出适当的反应。

    这个页面解释了被动模式是什么以及观察者和观察点是什么（以及观察者如何订阅观察点）。 其他页面显示如何使用各种观察点操作符将观察点链接在一起并更改其行为。

    背景¶

    在许多软件编程任务中，您或多或少地期望您编写的指令将按照您编写的顺序逐个执行并逐步完成。 但是在ReactiveX中，许多指令可能会并行执行，其结果稍后会以“观察者”的任意顺序捕获。 您不是调用某个方法，而是以“观察点”的形式定义一个检索和转换数据的机制，然后订阅一个观察者，此时，先前定义的机制触发观察员站岗,在准备就绪时捕捉和回应排放量。

    这种方法的一个优点是，当你有很多不相互依赖的任务时，你可以同时启动所有的任务，而不必等到每个任务完成后再开始下一个任务 - 这样，你的整个任务,捆绑任务只需要很长时间就可以完成捆绑中最长的任务。

    有很多术语用于描述这种异步编程和设计模型。 本文档将使用以下术语：观察者订阅观察点。 观察点通过调用观察者的方法来发射项目或向观察者发送通知。

    在其他文件和其他文件中，我们所称的“观察员”有时称为“用户”，“观察员”或“响应者”。 通常这种模式通常被称为“响应者模式”。

    建立观察员¶

    该页面使用类似Groovy的伪代码作为示例，但在许多语言中都有ReactiveX实现。

    在普通的方法调用中 - 也就是说，不是ReactiveX中典型的异步并行调用 - 流程如下所示：

    1. 调用一个方法。
    2. 将该方法的返回值存储在一个变量中。
    3. 使用该变量及其新值来做一些有用的事情。

    或者，像这样的东西：

    // 进行调用，将其返回值分配给`returnVal`
returnVal = someMethod(itsParameters);
// 用returnVal做一些有用的事情

    在异步模型中，流程更像这样：

    1. 定义一个方法，对异步调用的返回值做一些有用的操作;,这个方法是观察者的一部分。
    2. 将异步调用本身定义为观察点。
    3. 通过订阅将观察者附加到该观察点（这也启动观察点的行为）。
    4. 继续你的事业;,每当调用返回时，观察者的方法将开始以其返回值或值（由观察点发出的项目）进行操作。

    看起来像这样：

    // 看起来像这样：
// (在这个例子中，观察者非常简单，只有一个onNext处理程序)
def myOnNext = { it -> do something useful with it };
// 定义但不调用观察点
def myObservable = someObservable(itsParameters);
// 订阅订户到观察点，并调用观察点
myObservable.subscribe(myOnNext);
// 继续关于我的生意

    onNext, onCompleted, 和 onError¶

    Subscribe方法是您如何将观察者连接到观察点。 您的观察者实现了以下方法的一些子集：

    onNext

    只要观察点发射一个项目，观察点就会调用这个方法。 此方法将观察点发出的项目作为参数。

    onError

    观察点调用此方法来指示它未能生成预期数据或遇到其他一些错误。 它不会进一步调用onNext或onCompleted。 onError方法以其参数作为导致错误的指示。

    onCompleted

    如果观察点没有遇到任何错误，它会在最后一次调用onNext之后调用此方法。 根据观察点契约的条款，它可能会调用onNext零次或多次，然后可能会通过调用onCompleted或onError而不是两次，这将是它的最后一次调用。 按照惯例，在本文档中，对onNext的调用通常称为项目的“排放”，而对onCompleted或onError的调用称为“通知”。

    更完整的订阅调用示例如下所示：

    def myOnNext     = { item -> /* do something useful with item */ };
def myError      = { throwable -> /* react sensibly to a failed call */ };
def myComplete   = { /* clean up after the final response */ };
def myObservable = someMethod(itsParameters);
myObservable.subscribe(myOnNext, myError, myComplete);
// go on about my business

    退订¶

    在一些ReactiveX实现中，有一个专门的观察​​者接口Subscriber，它实现了一个取消订阅方法。 您可以调用此方法来指示订阅者不再对其当前订阅的任何观察对象感兴趣。 那些观察者可以（如果他们没有其他感兴趣的观察者）选择停止产生新的项目发射。

    这个取消订阅的结果将通过适用于观察者订阅的观察点的操作符链级联回来，这将导致链中的每个链接停止发送项目。 然而，这并不能保证立即发生，即使在没有观察者留下来观察这些排放的情况下，观察点也有可能产生并尝试排放物品。

    关于命名约定的一些注记¶

    ReactiveX的每个语言特定实现都有其自己的命名怪癖。 没有规范的命名标准，但实现之间有许多共同之处。

    此外，其中一些名称在其他情况下具有不同含义，或者在某种特定实施语言的习惯用语中看起来很尴尬。

    例如，有onEvent命名模式（例如onNext，onCompleted，onError）。 在某些情况下，这些名称将指示通过哪些事件处理程序注册的方法。 但是，在ReactiveX中，它们自己命名事件处理程序。

    “热”和“冷”观察物¶

    观察点什么时候开始发射它的序列？,这取决于观察点。 一个“热”观察点可能会在它创建后立即开始发布，因此任何稍后订阅该观察点的观察者都可能开始观察中间某处的序列。 另一方面，“冷”观察者等待观察者在开始发射物品之前订阅它，因此这样的观察者保证从头看到整个序列。

    在ReactiveX的一些实现中，还有一些称为“可连接”观察点的东西。 这样一个观察点不会开始发射项目，直到它的Connect方法被调用，无论观察者是否订阅了它。

    通过观察点操作符组合¶

    观察者和观察者只是ReactiveX的开始。 它们本身只不过是标准观察者模式的轻微扩展，更适合处理一系列事件而不是单个回调。

    真正的力量来自“反应性扩展”（因此称为“ReactiveX”） - 允许您转换，组合，操作和使用观察点发射的物品序列的操作符。

    这些Rx操作符允许您以声明方式一起组合异步序列，并具有回调的所有效率优势，但没有通常与异步系统关联的嵌套回调处理程序的缺点。

    操作链¶

    大多数操作员使用观察点进行操作并返回观察点。 这允许您将这些操作符一个接一个地应用在一个链中。 链中的每个操作符都会修改由前一个操作符的操作导致的观察点。

    还有其他模式，比如Builder模式，其中一个特定类的各种方法通过对该方法的操作修改该对象来操作该类的一个项目。 这些模式还允许您以类似的方式链接方法。 但是在Builder模式中，链中方法出现的顺序通常并不重要，观察点操作符排序很重要。

    一连串的观察点操作符并不独立于源自链的原始观察点，而是依次操作，每个操作都依赖于链中前一个操作符生成的观察点
