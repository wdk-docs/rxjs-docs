    观察体与其它技术的比较¶

    你可以经常使用观察体（Observable）而不是承诺（Promise）来异步传递值。 类似的，观察体也可以取代事件处理器的位置。最后，由于观察体传递多个值，所以你可以在任何可能构建和操作数组的地方使用观察体。

    在这些情况下，观察体的行为与其替代技术有一些差异，不过也提供了一些显著的优势。下面是对这些差异的详细比较。

    观察体 vs. 承诺¶

    观察体经常拿来和承诺进行对比。有一些关键的不同点：

    • 观察体是声明式的，在被订阅之前，它不会开始执行。承诺是在创建时就立即执行的。这让观察体可用于定义那些应该按需执行的菜谱。
    • 观察体能提供多个值。承诺只提供一个。这让观察体可用于随着时间的推移获取多个值。
    • 观察体会区分串联处理和订阅语句。承诺只有 .then() 语句。这让观察体可用于创建供系统的其它部分使用而不希望立即执行的复杂菜谱。
    • 观察体的 subscribe() 会负责处理错误。承诺会把错误推送给它的子承诺。这让观察体可用于进行集中式、可预测的错误处理。

    创建与订阅¶

    在有消费者订阅之前，观察体不会执行。subscribe() 会执行一次定义好的行为，并且可以再次调用它。每次订阅都是单独计算的。重新订阅会导致重新计算这些值。

    // declare a publishing operation
new Observable((observer) => { subscriber_fn });
// initiate execution
observable.subscribe(() => {
  // observer handles notifications
});

    承诺会立即执行，并且只执行一次。当承诺创建时，会立即计算出结果。没有办法重新做一次。所有的 then 语句（订阅）都会共享同一次计算。

    // initiate execution
new Promise((resolve, reject) => { executer_fn });
// handle return value
promise.then((value) => {
  // handle result here
});

    串联¶

    观察体会区分各种转换函数，比如映射和订阅。只有订阅才会激活订阅者函数，以开始计算那些值。

    observable.map((v) => 2*v);

    承诺并不区分最后的 .then() 语句（等价于订阅）和中间的 .then() 语句（等价于映射）。

    promise.then((v) => 2*v);

    可取消¶

    观察体的订阅是可取消的。取消订阅会移除监听器，使其不再接受将来的值，并通知订阅者函数取消正在进行的工作。

    const sub = obs.subscribe(...);
sub.unsubscribe();

    承诺是不可取消的。

    错误处理¶

    观察体的错误处理是交给订阅者的错误处理器的，并且该订阅者会自动取消对这个观察体的订阅。

    obs.subscribe(() => {
  throw Error('my error');
});

    承诺会把错误推给其子承诺。

    promise.then(() => {
  throw Error('my error');
});

    速查表¶

    下列代码片段揭示了同样的操作要如何分别使用观察体和承诺进行实现。

    操作 观察体 承诺
    创建 new Observable((observer) => { observer.next(123);}); new Promise((resolve, reject) => {resolve(123);});
    转换 obs.map((value) => value * 2 ); promise.then((value) => value * 2);
    订阅 sub = obs.subscribe((value) => {console.log(value)}); promise.then((value) => {console.log(value);});
    取消订阅 sub.unsubscribe(); 承诺被解析时隐式完成。

    观察体 vs. 事件 API¶

    观察体和事件 API 中的事件处理器很像。这两种技术都会定义通知处理器，并使用它们来处理一段时间内传递的多个值。订阅观察体与添加事件处理器是等价的。一个显著的不同是你可以配置观察体，使其在把事件传给事件处理器之间先进行转换。

    使用观察体来处理错误和异步操作在 HTTP 请求这样的场景下更加具有一致性。

    下列代码片段揭示了同样的操作要如何分别使用观察体和事件 API 进行实现。

    创建与取消¶

    观察体¶

    // Setup
let clicks$ = fromEvent(buttonEl, ‘click’);
// Begin listening
let subscription = clicks$
  .subscribe(e => console.log(‘Clicked’, e))
// Stop listening
subscription.unsubscribe();

    事件¶

    function handler(e) {
  console.log(‘Clicked’, e);
}

// Setup & begin listening
button.addEventListener(‘click’, handler);
// Stop listening
button.removeEventListener(‘click’, handler);

    订阅¶

    观察体¶

    observable.subscribe(() => {
  // notification handlers here
});

    事件¶

    element.addEventListener(eventName, (event) => {
  // notification handler here
});

    配置¶

    观察体¶

    监听按键，提供一个流来表示这些输入的值。

    fromEvent(inputEl, 'keydown').pipe(
  map(e => e.target.value)
);

    事件¶

    不支持配置。

    element.addEventListener(eventName, (event) => {
  // Cannot change the passed Event into another
  // value before it gets to the handler
});

    观察体 vs. 数组¶
